Návrh databází: Od konceptu po realizaci

Úvod

Návrh databáze je klíčovou fází vývoje softwaru. Špatně navržená databáze vede k redundanci (opakování) dat, nekonzistenci (různé údaje o téže věci na různých místech) a pomalému výkonu. Cílem návrhu je vytvořit strukturu, která efektivně ukládá data a zachovává jejich integritu.

Proces návrhu dělíme na tři hlavní fáze:

  1. Konceptuální návrh (ER diagramy)
  2. Logický návrh (Normalizace, relační schéma)
  3. Fyzický návrh (SQL kód)

Praktická ukázka: Databáze pro E-shop

Pro demonstraci budeme navrhovat zjednodušený systém pro e-shop.

Zadání: Potřebujeme evidovat zákazníky, produkty a objednávky. Víme, že jedna objednávka může obsahovat více produktů a jeden produkt může být ve více objednávkách.

Fáze 1: Konceptuální model (ER Diagram)

V této fázi nás nezajímají tabulky ani datové typy, ale entity (objekty) a vztahy mezi nimi.

Identifikované entity:

  • Zákazník (Jméno, Email, Adresa)
  • Produkt (Název, Cena)
  • Objednávka (Datum, Stav)

Vztahy:

  1. Zákazník vytváří Objednávku (Vztah 1:N – jeden zákazník může mít mnoho objednávek).
  2. Objednávka obsahuje Produkty (Vztah M:N – mnoho produktů v mnoha objednávkách).

Poznámka: Vztah M:N (Mnoho k mnoha) je v relačních databázích problematický a v další fázi jej budeme muset rozbít pomocí tzv. vazební tabulky.

Fáze 2: Logický návrh a Normalizace

Zde převádíme entity na tabulky a aplikujeme pravidla Normalizace. Normalizace je proces odstraňování redundance.

Třetí normální forma (3NF) - Prakticky

Představme si, že bychom vše nacpali do jedné tabulky. To by byla chyba:

ID_ObjZákazníkAdresaProdukt
101Jan NovákPraha 1Notebook
102Jan NovákPraha 1Myš

Problém: Jméno "Jan Novák" a adresa "Praha 1" se opakují. Pokud se Jan přestěhuje, musíme to měnit na 100 místech. To porušuje principy normalizace.

Řešení (Rozdělení do tabulek):

  1. Tabulka Zakaznici (uložíme Jana jen jednou).
  2. Tabulka Produkty.
  3. Tabulka Objednavky (odkazuje na Jana pomocí ID).
  4. Tabulka Polozky_Objednavky (řeší vztah M:N – propojuje objednávku a produkt).

Fáze 3: Fyzický návrh (SQL Kód)

Toto je finální kód, který spustíme v databázi (např. MySQL, PostgreSQL). Zde definujeme datové typy (INT, VARCHAR, DECIMAL) a klíče.

Primární klíč (PK): Jedinečný identifikátor řádku.

Cizí klíč (FK): Odkaz na primární klíč jiné tabulky (vytváří vztah).

-- 1. Tabulka Zákazníků
CREATE TABLE Zakaznici (
    zakaznik_id INT PRIMARY KEY AUTO_INCREMENT,
    jmeno VARCHAR(100) NOT NULL,
    email VARCHAR(100) UNIQUE NOT NULL,
    adresa VARCHAR(255)
);

-- 2. Tabulka Produktů
CREATE TABLE Produkty (
    produkt_id INT PRIMARY KEY AUTO_INCREMENT,
    nazev VARCHAR(100) NOT NULL,
    cena DECIMAL(10, 2) NOT NULL -- Cena s 2 desetinnými místy
);

-- 3. Tabulka Objednávek
CREATE TABLE Objednavky (
    objednavka_id INT PRIMARY KEY AUTO_INCREMENT,
    zakaznik_id INT, -- Cizí klíč
    datum_vytvoreni DATETIME DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (zakaznik_id) REFERENCES Zakaznici(zakaznik_id)
);

-- 4. Vazební tabulka (řeší vztah M:N)
CREATE TABLE Polozky_Objednavky (
    id INT PRIMARY KEY AUTO_INCREMENT,
    objednavka_id INT,
    produkt_id INT,
    pocet_kusu INT DEFAULT 1,
    FOREIGN KEY (objednavka_id) REFERENCES Objednavky(objednavka_id),
    FOREIGN KEY (produkt_id) REFERENCES Produkty(produkt_id)
);

Vysvětlení kódu: V tabulce Objednavky neukládáme jméno zákazníka, ale pouze zakaznik_id. Pokud se Jan Novák přejmenuje, změníme to jen v tabulce Zakaznici a všechny jeho objednávky zůstanou platné. To je síla relační databáze.

Tabulka Polozky_Objednavky nám umožňuje říct: "V objednávce č. 1 je 2x Produkt A a 1x Produkt B".

Závěr

Kvalitní návrh databáze připomíná stavbu domu.

  1. Nejdřív potřebujeme nákres (ER Diagram).
  2. Pak musíme rozvrhnout místnosti tak, aby se v domě dalo žít a nebylo v něm zbytečné místo (Normalizace).
  3. Nakonec dům postavíme z cihel (SQL kód).

Dodržení tohoto postupu zajišťuje, že aplikace bude stabilní, data budou v bezpečí a systém bude možné v budoucnu snadno rozšiřovat.
